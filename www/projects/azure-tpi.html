Infrastructure Azure AD

TPI ETML 2021 : Évaluation et comparaison de deux architectures Azure AD (hybride vs cloud) pour l'enseignement. 5 semaines, 90 heures.

Contexte

Travail Pratique Individuel (TPI) réalisé à l'ETML en mai–juin 2021 (classe CIN4A). L'objectif est d'évaluer la migration d'une infrastructure de labo scolaire vers Microsoft Azure Active Directory, et d'en comparer deux scénarios types : une solution hybride (VMs locales + Azure AD) et une solution 100% cloud (Azure VMs + Azure AD via RBAC). Chaque scénario fait l'objet d'une analyse financière détaillée. Le tout est documenté pour servir de base de cours cloud à l'école.

AD · Azure AD · Azure ADDS

Une partie analyse du rapport est consacrée à la différenciation entre les trois services Microsoft, souvent confondus :

Active Directory (AD) LDAP on-premise, gestion des objets (utilisateurs, groupes, OU), GPO, arbres/forêts/domaines. Authentification Kerberos/NTLM.
Azure Active Directory (AAD) Service cloud-only. Pas de GPO, pas de LDAP/Kerberos/NTLM, pas d'arbre/forêt. Authentification OAuth2/SAML. Parfait pour SSO SaaS (Office 365, Dynamics 365).
Azure ADDS Étend l'AAD avec LDAP, Kerberos, GPO et des contrôleurs de domaine managés. Permet la synchronisation bidirectionnelle avec un AD on-premise via Azure AD Connect.
Azure Bastion PaaS, accès RDP/SSH aux VMs Azure via le navigateur sur le port HTTPS 443. Évite d'exposer le port 3389 sur internet. Coût additionnel par rapport aux VMs seules.

Problèmes rencontrés & solutions

Trois obstacles techniques majeurs ont nécessité des adaptations en cours de projet :

Crédits liés au tenant par défaut Les 100 $ de crédits étudiant sont liés au répertoire par défaut et ne peuvent pas être transférés vers un autre tenant. La VM Azure et l'Azure AD doivent donc coexister dans le même tenant.
Azure Bastion ≠ Azure AD auth Azure Bastion ne supporte pas l'authentification Azure AD (AADLoginForWindows), seul RDP direct est pris en charge. Solution : passer à RDP direct, en utilisant le partage de connexion du téléphone pour contourner le blocage du port 3389 par le réseau de l'école.
NLA bloque la connexion AAD via RDP La connexion RDP avec un compte Azure AD échoue si NLA (Network Level Authentication) est actif. Il faut désactiver NLA sur la VM et injecter des paramètres spécifiques dans le fichier .rdp pour contourner la pop-up de credentials.
connexion-azure-ad.rdp RDP 
# Paramètres requis pour connexion RDP avec compte Azure AD (NLA désactivé)
full address:s:51.107.86.22:3389
prompt for credentials:i:0
authentication level:i:2
enablecredsspsupport:i:0       # désactive NLA côté client
username:s:AzureAD\Eleve_C@jesuisunetudiantoutlook.onmicrosoft.com
domain:s:AzureAD

Infrastructure déployée

Resource Groups Scenario_N1 et Scenario_N2 dans la région Suisse Nord, isolation des ressources et suivi des coûts par scénario.
2 tenants Azure AD ScenarioNumero1.onmicrosoft.com (scénario hybride) et jesuisunetudiant.onmicrosoft.com (scénario cloud RBAC).
Azure VNet Réseau virtuel avec 4 sous-réseaux pour le scénario 2. Sous-réseau AzureBastionSubnet dédié. NSG pour contrôle du trafic entrant/sortant.
Azure VM Windows 10 Plan minimal (1 vCPU, 0.5 GB RAM, environ 4.48 CHF/mois) suffisant pour les tests d'authentification. Connexion via RDP 3389 depuis hotspot mobile (réseau école bloque 3389).
RBAC sur la VM Rôles Virtual Machine User Login et Virtual Machine Administrator Login assignés aux utilisateurs/administrateurs de l'Azure AD pour contrôle d'accès fin.
Power Automate backup Flux automatisé pour sauvegardes journalières de la documentation vers OneDrive et Google Drive. Nommage automatique des dossiers par date.

Résultats

Tous les tests du dossier de tests ont été concluants sur les deux scénarios :

  • Azure AD scénario 1 (hybride) : tenant opérationnel, ajout/suppression d'utilisateurs et groupes fonctionnel
  • VMs VirtualBox jointes au domaine Azure AD, connexion avec 3 comptes utilisateurs différents validée
  • Azure AD scénario 2 (RBAC) : VM Azure jointe au tenant, connexion RDP avec compte Azure AD confirmée
  • Accès réseau inter-VMs via ICMP validé, accès internet confirmé depuis les VMs Azure
  • Scénario 1 : coût total 0 CHF. Crédits $100 intacts après le TPI
