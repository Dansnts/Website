TurboDex

Pokédex pour voitures. App mobile Flutter, infra Azure IaC (Terraform/Ansible/K8s). PDG 2025 HEIG-VD

Contexte

Projet de groupe (4 personnes) réalisé en 2025 à la HEIG-VD. TurboDex est un Pokédex pour passionnés d'automobiles : photographier un véhicule, l'IA identifie la marque/modèle, et la voiture rejoint ta collection. Mon rôle : DevOps, infrastructure Azure complète en IaC, CI/CD, et déploiement des microservices IA sur AKS.

Application

Interface Flutter cross-platform (iOS/Android) et feature de floutage automatique des visages et plaques d'immatriculation (privacy-first).

Architecture Azure

Infrastructure as Code : Terraform

L'ensemble de l'infrastructure Azure est déclarée en HCL. Un terraform apply provisionne le Resource Group, l'ACR, l'AKS, la Cosmos DB MongoDB, le Blob Storage et la Function App.

Azureinfra/terraform/main.tf HCL 
# AKS Cluster — SystemAssigned identity
resource "azurerm_kubernetes_cluster" "aks" {
  name                = "${var.project_name}-${var.environment}-aks"
  location            = azurerm_resource_group.rg.location
  resource_group_name = azurerm_resource_group.rg.name
  dns_prefix          = "${var.project_name}-${var.environment}"

  default_node_pool {
    name       = "system"
    node_count = var.aks_node_count   # default 2
    vm_size    = var.aks_vm_size      # Standard_DS2_v2
  }
  identity { type = "SystemAssigned" }
}

# Cosmos DB MongoDB — Session consistency, shard sur _id
resource "azurerm_cosmosdb_account" "cosmos" {
  name        = local.cosmos_account_name
  kind        = "MongoDB"
  offer_type  = "Standard"

  capabilities       { name = "EnableMongo" }
  consistency_policy { consistency_level = "Session" }
  geo_location       { location = azurerm_resource_group.rg.location; failover_priority = 0 }
}

resource "azurerm_cosmosdb_mongo_collection" "collection" {
  throughput = 400
  shard_key  = "_id"
  index      { keys = ["_id"] }
}

# Function App Linux Consumption (Y1) + Application Insights
resource "azurerm_linux_function_app" "func" {
  https_only  = true
  identity    { type = "SystemAssigned" }

  site_config {
    application_stack { python_version = "3.11" }
    cors              { allowed_origins = ["*"] }
  }

  app_settings = {
    APPINSIGHTS_INSTRUMENTATIONKEY        = azurerm_application_insights.appi.instrumentation_key
    COSMOS_CONNECTION_STRING              = azurerm_cosmosdb_account.cosmos.connection_strings[0]
    BLOB_CONNECTION_STRING                = azurerm_storage_account.sa.primary_connection_string
    FUNCTIONS_WORKER_RUNTIME              = var.function_runtime
  }
}

resource "azurerm_service_plan" "func_plan" {
  os_type  = "Linux"
  sku_name = "Y1"   # Consumption plan
}

Déploiement K8s des microservices IA

Les deux microservices IA (blur et analyse) sont déployés sur AKS via Terraform. Les secrets ACR et Cosmos DB sont injectés depuis un Kubernetes Secret.

Azureinfra/terraform/k8s.tf HCL 
# Secret K8s — ACR + Cosmos DB + Blob Storage
resource "kubernetes_secret" "app_secrets" {
  metadata { name = "app-secrets"; namespace = "default" }
  data = {
    DOCKER_REGISTRY_SERVER_URL      = var.acr_login_server
    DOCKER_REGISTRY_SERVER_USERNAME = azurerm_container_registry.acr.admin_username
    DOCKER_REGISTRY_SERVER_PASSWORD = azurerm_container_registry.acr.admin_password
    COSMOS_CONNECTION_STRING         = azurerm_cosmosdb_account.cosmos.connection_strings[0]
    BLOB_CONNECTION_STRING           = azurerm_storage_account.sa.primary_connection_string
  }
}

# Deployment Blur IA — 2 replicas, image depuis ACR
resource "kubernetes_deployment" "blur" {
  spec {
    replicas = 2
    template {
      spec {
        container {
          name  = "blur"
          image = "${var.acr_login_server}/${var.blur_image_name}:${var.blur_image_tag}"
          env_from {
            secret_ref { name = kubernetes_secret.app_secrets.metadata[0].name }
          }
        }
      }
    }
  }
}

# LoadBalancer public pour le service Blur
resource "kubernetes_service" "blur_lb" {
  spec {
    type             = "LoadBalancer"
    selector         = { app = "blur" }
    load_balancer_ip = var.blur_lb_ip   # IP fixe optionnelle
    port { port = 80; target_port = var.blur_container_port }
  }
}

# Deployment Analyse IA — même pattern, port 5000
resource "kubernetes_deployment" "analyse" {
  spec {
    replicas = 2
    template {
      spec {
        container {
          image = "${var.acr_login_server}/${var.analyse_image_name}:${var.analyse_image_tag}"
          env_from {
            secret_ref { name = kubernetes_secret.app_secrets.metadata[0].name }
          }
        }
      }
    }
  }
}

CI/CD : GitHub Actions

Le pipeline Terraform tourne sur chaque push. L'apply est déclenché manuellement uniquement (workflow_dispatch) pour éviter tout provisioning involontaire. Ansible s'exécute après Terraform en dry-run (--check).

.github/workflows/infra.yml YAML 
name: Infra CI/CD
on:
  push:    { branches: [main] }
  pull_request: { branches: [main] }
  workflow_dispatch:   # apply uniquement si déclenché manuellement

jobs:
  terraform:
    runs-on: ubuntu-latest
    defaults:
      run:
        working-directory: ./Azureinfra/terraform
    env:
      ARM_CLIENT_ID:       ${{ secrets.ARM_CLIENT_ID }}
      ARM_CLIENT_SECRET:   ${{ secrets.ARM_CLIENT_SECRET }}
      ARM_SUBSCRIPTION_ID: ${{ secrets.ARM_SUBSCRIPTION_ID }}
      ARM_TENANT_ID:       ${{ secrets.ARM_TENANT_ID }}
    steps:
      - uses: actions/checkout@v4
      - uses: hashicorp/setup-terraform@v3
        with:
          terraform_version: 1.9.8
      - run: terraform init
      - run: terraform validate
      - run: terraform plan -out=tfplan
      - name: Terraform Apply
        if: github.event_name == 'workflow_dispatch'
        run: terraform apply -auto-approve tfplan

  ansible:
    needs: terraform
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: sudo apt-get install -y ansible
      - run: ansible-playbook Azureinfra/ansible/playbooks/site.yml
              -i Azureinfra/ansible/inventory.ini --check  # dry-run

DevOps : Build, Test & Push

Pipeline dédié pour les microservices IA : l'image Docker est buildée, les tests pytest tournent dans le container, et le push vers l'ACR n'a lieu que si les tests passent. Seuls les merges sur main déclenchent le push.

.github/workflows/docker-build.yml YAML 
name: Docker Build & Push

on:
  push:         { branches: [main, dev] }
  pull_request: { branches: [main] }

env:
  REGISTRY:   turbodexacr.azurecr.io
  IMAGE_NAME: backend

jobs:
  build-test-push:
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v4

      # 1 — Build
      - name: Build image
        run: |
          docker build \
            -t ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:${{ github.sha }} \
            -t ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:latest .

      # 2 — Test (pytest s'exécute dans le container)
      - name: Run tests
        run: |
          docker run --rm \
            -e TESTING=true \
            ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:${{ github.sha }} \
            pytest tests/ -v --tb=short

      # 3 — Push uniquement si tests OK + merge sur main
      - name: Login to ACR
        if: github.ref == 'refs/heads/main' && github.event_name == 'push'
        uses: azure/docker-login@v1
        with:
          login-server: ${{ env.REGISTRY }}
          username:     ${{ secrets.ACR_USERNAME }}
          password:     ${{ secrets.ACR_PASSWORD }}

      - name: Push to ACR
        if: github.ref == 'refs/heads/main' && github.event_name == 'push'
        run: |
          docker push ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:${{ github.sha }}
          docker push ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:latest
