CrunchTime, Ligue Pulmonaire Vaudoise

Infrastructure K8s on-premise pour la classification automatique de mails par IA. Sprint d'innovation HEIG-VD 2026.

Contexte

CrunchTime Innovation est un challenge hebdomadaire organisé par la HEIG-VD : cinq jours pour comprendre un mandat client réel, concevoir une solution, prototyper et pitcher. En mars 2026, notre équipe de 5 étudiants interdisciplinaires a travaillé pour la Ligue Pulmonaire Vaudoise (LPV), association lausannoise de 80 collaborateurs spécialisée dans les maladies respiratoires. Mon rôle : conception et design de l'infrastructure Kubernetes on-premise proposée au client.

Le mandat

Comment mettre en place un système de tri et d'aiguillage automatique des emails entrants sur info@lpvd.ch, capable de les classifier et de les redistribuer vers les bonnes boîtes selon leur nature (demandes médicales, administratives ou financières), tout en garantissant la protection des données médicales sensibles ?

Le problème

La LPV gère une boîte mail partagée avec une quinzaine de sous-boîtes internes. Chaque email entrant doit être traité manuellement : lire le contenu, ouvrir les pièces jointes, comprendre de quoi il s'agit, puis rediriger vers le bon service. Dans un contexte médical, une erreur de routage n'est pas un simple colis perdu.

Charge cognitive élevée

Lire, comprendre et trier chaque email manuellement plusieurs dizaines de fois par jour.

Risque d'erreur élevé

Un email mal redirigé dans un contexte médical peut avoir des conséquences sérieuses.

Données médicales sensibles

Toute solution doit respecter la nLPD, aucune donnée ne peut transiter vers un service cloud externe.

Schéma logique

Le processus de traitement a été défini en cinq étapes lors de la phase d'idéation, avant de concevoir l'architecture technique.

1 Étape

Réception

Email entrant sur info@lpvd.ch, réception via Microsoft Outlook

2 Étape

Analyse du contenu

Branche A Corps du mail

Extraction du texte → analyse NLP → LLM

Branche B Pièces jointes PDF

PDF parser → extraction du texte → LLM

3 Étape

Classification

Attribution d'une catégorie : Prescriptions médicales · Rapports médicaux · Administratif · Finance · Thérapie · Autres

4 Étape

Aiguillage automatique

Routage vers la sous-boîte concernée : Finance · Médicale · Administration · Prescriptions…

5 Étape

Validation humaine

L'admin valide ou corrige. La correction est enregistrée et réinjectée dans le modèle, boucle d'amélioration continue.

Processus technique : Architecture K8s

Après la définition du schéma logique, j'ai conçu l'architecture technique on-premise. Le cluster Kubernetes orchestre tous les microservices : un pod Nginx fait office de gateway, deux pods IA traitent les mails en parallèle (LLM + algorithme Python), PostgreSQL trace chaque action, et MinIO stocke les poids des modèles. GitLab CI/CD + Docker gère le déploiement à la volée.

Mon rôle : Infrastructure Kubernetes

En tant qu'ingénieur réseaux et systèmes de l'équipe, j'ai pris en charge la conception complète de l'infrastructure technique proposée au client. Le choix de Kubernetes on-premise s'est imposé pour trois raisons : la contrainte nLPD (données médicales, aucun cloud externe), la gestion native des ressources GPU NVIDIA pour l'inférence LLM, et la capacité à déployer ou mettre à jour les pods IA à la volée via GitLab CI/CD.

Kubernetes on-premise Orchestration des pods IA, gestion du plugin NVIDIA GPU, montée en charge automatique.
Nginx API Gateway Point d'entrée unique du cluster, extraction des données mail et fan-out vers les pods d'analyse.
LLM Microservice (pod) Modèle de langage isolé dans son pod, analyse sémantique du corps et des pièces jointes.
Python Classifier (pod) Algorithme de classification par règles métier, résultat déterministe pour audit et traçabilité.
PostgreSQL Traçabilité complète de chaque email traité, base de conformité nLPD.
MinIO Stockage objet on-premise pour les poids et versions des modèles IA, aucune donnée en cloud.
GitLab CI/CD + Docker Pipeline de déploiement à la volée, mise à jour des modèles ou de la configuration sans interruption.
Next.js Dashboard Interface de monitoring temps réel : métriques, alertes, feedback loop « Corriger & apprendre ».

100% On-Premise

Contrainte non négociable : la LPV traite des données médicales couvertes par la nLPD. Aucune donnée (emails, pièces jointes, poids de modèle) ne transite par un cloud externe. Tout s'exécute sur l'infrastructure locale hospitalière. C'est pourquoi MinIO remplace S3 et GitLab self-hosted remplace GitHub Actions.
